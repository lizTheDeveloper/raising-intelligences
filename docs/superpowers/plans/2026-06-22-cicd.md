# CI/CD + Infrastructure Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Auto-deploy Raising Intelligences to Hetzner on every merge to `main`, gated on passing tests, using Coolify (already running on the server) as the CI/CD platform.

**Architecture:** A multi-stage Dockerfile builds the React client then the TypeScript server, runs tests as a build gate, and produces a single production image that serves both static files and the API. Coolify connects to GitHub, triggers on push to `main`, and Traefik routes `multiversegames.ai` to the container automatically. Langfuse v2 is deployed as a separate Coolify Docker Compose stack for LLM observability.

**Tech Stack:** Docker multi-stage build, Coolify (self-hosted PaaS), Traefik (reverse proxy, already running), Langfuse v2 (self-hosted observability), Node.js 20 Alpine, npm workspaces, Socket.io (multiplayer transport — Traefik passes WebSocket upgrades automatically).

## Global Constraints

- Node.js base image: `node:20-alpine` only
- Langfuse: v2 only (not v3 — v3 requires ClickHouse which is too heavy for this server)
- Tests must gate deployment: if `npm run test -w server` exits non-zero, the Docker build fails and nothing is deployed
- The game DB and Langfuse DB must be separate Postgres instances
- `db/docker-compose.yml` is for local dev only — do not use it in production config
- All secrets go in Coolify's environment variable UI, never in the Dockerfile or committed files
- The server env var for LLM is `OPENROUTER_API_KEY` (even though the current PR uses `@anthropic-ai/sdk` — set it now, swap the client later)

## Task 1: Add static file serving to the server

The server currently has no middleware to serve the React client. Without this, the Dockerfile's production image would serve only the API — the frontend would be unreachable.

**Files:**
- Modify: `server/src/index.ts`

**Interfaces:**
- Produces: `GET /` → serves `client/dist/index.html` when `NODE_ENV=production`
- Produces: `GET /assets/*` → serves Vite-built static assets
- Existing: `GET /api/*` routes are unaffected (registered before static middleware)
- Existing: `GET /health` is unaffected
- Existing: Socket.io WebSocket handshake at `GET /socket.io/` is unaffected (handled by `httpServer`, not `app` routes)

> No test file needed here — `express.static()` is a single well-understood call and the smoke test in Task 2 Step 4 provides the real verification. YAGNI.

- [ ] **Step 1: Add static file middleware to `server/src/index.ts`**

Add `import path from "path";` to the existing imports at the top of the file. Then inside `main()`, insert the static serving block **after** `app.get("/health", ...)` and **before** the socket.io setup:

```typescript
import path from "path";
```

Inside `main()`, the section around the health route should become:

```typescript
  app.get("/health", (_req, res) => {
    res.json({ status: "ok" });
  });

  if (process.env.NODE_ENV === "production") {
    const clientDist = path.join(process.cwd(), "client", "dist");
    app.use(express.static(clientDist));
    app.get("*", (_req, res) => {
      res.sendFile(path.join(clientDist, "index.html"));
    });
  }

  // Realtime multiplayer transport (M2/M3). The REST routes above remain for
  // solo play and reconnect; socket.io drives two-player games over the same
  // engines, games Map, and repository.
  const sessions = new Map<string, Session>();
  const httpServer = createServer(app);
  const io = new SocketServer(httpServer, { cors: { origin: "*" } });
```

The catch-all `app.get("*", ...)` must come after all API and health routes so it only fires for unknown paths (React handles client-side routing from there). Socket.io mounts on `httpServer` separately and is not affected by Express middleware order.

- [ ] **Step 2: Run all server tests to confirm nothing broke**

```bash
npm run test -w server
```

Expected: 50 tests pass across 8 files.

- [ ] **Step 3: Commit**

```bash
git add server/src/index.ts
git commit -m "feat: serve React client static files in production"
```

## Task 2: Dockerfile + .dockerignore

A multi-stage Docker build that:
1. Runs tests — if they fail, the build fails and nothing deploys
2. Builds the React client
3. Compiles the TypeScript server
4. Assembles a minimal production image

**Files:**
- Create: `Dockerfile`
- Create: `.dockerignore`

**Interfaces:**
- Produces: Docker image that starts with `node server/dist/index.js`
- Produces: `GET /health` → `{"status":"ok"}` (smoke test target)
- Produces: `GET /` → React app index.html
- Consumes: `NODE_ENV`, `PORT`, `DATABASE_URL`, `OPENROUTER_API_KEY`, `LANGFUSE_HOST`, `LANGFUSE_PUBLIC_KEY`, `LANGFUSE_SECRET_KEY` at runtime (not baked in)

- [ ] **Step 1: Create `.dockerignore`**

```
node_modules
client/node_modules
server/node_modules
.git
.env*
db/
docs/
*.log
```

- [ ] **Step 2: Create `Dockerfile`**

```dockerfile
# ── Stage 1: run tests ────────────────────────────────────────────────────────
FROM node:20-alpine AS test
WORKDIR /app
COPY package.json package-lock.json tsconfig.base.json ./
COPY client/package.json ./client/
COPY server/package.json ./server/
RUN npm ci
COPY server/ ./server/
RUN npm run test -w server

# ── Stage 2: build client ─────────────────────────────────────────────────────
FROM node:20-alpine AS build-client
WORKDIR /app
COPY package.json package-lock.json tsconfig.base.json ./
COPY client/package.json ./client/
COPY server/package.json ./server/
RUN npm ci
COPY client/ ./client/
RUN npm run build -w client

# ── Stage 3: build server ─────────────────────────────────────────────────────
FROM node:20-alpine AS build-server
WORKDIR /app
COPY package.json package-lock.json tsconfig.base.json ./
COPY client/package.json ./client/
COPY server/package.json ./server/
RUN npm ci
COPY server/ ./server/
RUN npm run build -w server

# ── Stage 4: production image ─────────────────────────────────────────────────
FROM node:20-alpine AS production
WORKDIR /app
COPY package.json package-lock.json ./
COPY client/package.json ./client/
COPY server/package.json ./server/
RUN npm ci --omit=dev
COPY --from=build-server /app/server/dist ./server/dist
COPY --from=build-client /app/client/dist ./client/dist
ENV NODE_ENV=production
EXPOSE 3000
CMD ["node", "server/dist/index.js"]
```

> The `test` stage runs before all other stages. If `vitest run` exits non-zero, the `docker build` command fails and Coolify aborts — the running container is untouched.

- [ ] **Step 3: Build the image locally**

```bash
docker build -t ri-local .
```

Expected output ends with:
```
=> exporting to image
=> => naming to docker.io/library/ri-local
```

If the test stage fails you'll see vitest output and a non-zero exit. Fix failing tests before proceeding.

- [ ] **Step 4: Start the container and smoke-test it**

```bash
docker run --rm -d -p 3001:3000 --name ri-smoke -e NODE_ENV=production ri-local
sleep 2
curl http://localhost:3001/health
curl -s -o /dev/null -w "%{http_code}" http://localhost:3001/
docker stop ri-smoke
```

Expected:
- `curl /health` → `{"status":"ok"}`
- `curl /` → HTTP `200` (React app's index.html)

If `/` returns 404, the `client/dist` copy path in the Dockerfile or the `path.join(process.cwd(), "client", "dist")` in `index.ts` is wrong. The WORKDIR is `/app`, so `process.cwd()` = `/app` and the full path is `/app/client/dist`.

```bash

git commit -m "feat: multi-stage Dockerfile with test gate"
```

## Task 4: Langfuse v2 on Coolify

Deploy a self-hosted Langfuse v2 instance for LLM observability. Langfuse v2 requires only Postgres (no Redis, no ClickHouse). Deploy it as a Docker Compose stack in Coolify.

**Files:**
- Create: `langfuse/docker-compose.yml` (checked in for reference, deployed via Coolify UI)

**Interfaces:**
- Produces: `https://langfuse.multiversegames.ai` → Langfuse UI
- Produces: project API keys to paste into game app env vars

- [ ] **Step 1: Generate secrets**

Run locally:

```bash
openssl rand -base64 32  # copy this → NEXTAUTH_SECRET
openssl rand -base64 32  # copy this → SALT
```

Save these somewhere secure (1Password, etc.) before continuing.

- [ ] **Step 2: Create the Langfuse docker-compose file**

Create `langfuse/docker-compose.yml` in the repo (for reference and reproducibility):

```yaml
services:
  langfuse:
    image: ghcr.io/langfuse/langfuse:2
    restart: unless-stopped
    depends_on:
      langfuse-db:
        condition: service_healthy
    ports:
      - "3001:3000"
    environment:
      DATABASE_URL: postgresql://langfuse:${LANGFUSE_DB_PASSWORD}@langfuse-db:5432/langfuse
      NEXTAUTH_URL: https://langfuse.multiversegames.ai
      NEXTAUTH_SECRET: ${NEXTAUTH_SECRET}
      SALT: ${SALT}
      TELEMETRY_ENABLED: "false"

  langfuse-db:
    image: postgres:15-alpine
    restart: unless-stopped
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U langfuse"]
      interval: 3s
      timeout: 3s
      retries: 10
    environment:
      POSTGRES_DB: langfuse
      POSTGRES_USER: langfuse
      POSTGRES_PASSWORD: ${LANGFUSE_DB_PASSWORD}
    volumes:
      - langfuse_db:/var/lib/postgresql/data

volumes:
  langfuse_db:
```

- [ ] **Step 3: Deploy via Coolify**

In Coolify: **Applications → Add New Application → Docker Compose**

- Paste the contents of `langfuse/docker-compose.yml`
- Set environment variables in Coolify's UI for this stack:
  - `NEXTAUTH_SECRET` = value from Step 1
  - `SALT` = value from Step 1
  - `LANGFUSE_DB_PASSWORD` = any strong random password (generate with `openssl rand -hex 16`)
- Domain: `langfuse.multiversegames.ai` (assign to the `langfuse` service, port `3000`)

Click **Deploy** and wait for both containers to start.

- [ ] **Step 4: Create Langfuse project and get API keys**

Open `https://langfuse.multiversegames.ai` in a browser.

1. Sign up for an account (first user becomes admin)
2. Create a new project: **Raising Intelligences**
3. Go to **Settings → API Keys → Create new API key**
4. Copy the **Public Key** and **Secret Key**

- [ ] **Step 5: Wire Langfuse keys into the game app**

In Coolify, open the **Raising Intelligences** app → **Environment Variables**:

- Set `LANGFUSE_PUBLIC_KEY` = the public key from Step 4
- Set `LANGFUSE_SECRET_KEY` = the secret key from Step 4
- Set `LANGFUSE_HOST` = `https://langfuse.multiversegames.ai` (should already be set from Task 3)

Click **Redeploy**.

- [ ] **Step 6: Verify traces appear in Langfuse**

1. Open `https://multiversegames.ai` and play through one event (start a game, send a message to the kid)
2. Open `https://langfuse.multiversegames.ai` → **Traces**

Expected: LLM calls appear as traces within a few seconds. Each call shows model, tokens, latency.

If no traces appear: check the game app logs in Coolify for Langfuse connection errors. The `isLangfuseEnabled()` function in `server/src/observability/langfuse.ts` returns true only when all three Langfuse env vars are set.

- [ ] **Step 7: Commit the docker-compose reference file**

```bash
git add langfuse/docker-compose.yml
git commit -m "infra: add Langfuse v2 docker-compose reference for Coolify"
```
